# 仕様復元(コード考古学)

## 目的

整備されたリファレンスドキュメントが存在しない独自システムについて、利用可能な一次情報
(ソースコード、システムが吐くログ、オペレータの日誌)から仕様を復元し、
以後の自動化・保守の土台となるリファレンス文書を作成する。

## 手順

1. **情報源の棚卸し**: 対象システムについて入手できる情報源(コード、ログ、日誌、設定ファイル、
   スキーマ)を列挙し、それぞれの網羅性・鮮度を評価する。
2. **外形仕様の抽出**(コードがある場合):
   - エントリポイント、外部インターフェース(API/CLI/ファイル入出力/DB)を特定する
   - 入出力のデータ形式・バリデーション・エラー処理を読み取る
   - 設定項目とデフォルト値を列挙する
3. **挙動仕様の抽出**(ログがある場合): ログメッセージの種類からシステムの状態遷移・
   処理フロー・異常系の挙動を復元する。コード内のログ出力箇所と突き合わせると精度が上がる。
4. **運用仕様の抽出**(日誌がある場合): 実際の運用手順・暗黙のルール・既知の癖(ワークアラウンド)
   を吸い上げる。文書化されていない運用知識こそ最優先で残す。
5. **確度の区分け**: 復元した各仕様に確度を付す。
   - 【確実】コードで確認できた
   - 【推定】ログ・日誌からの推定(根拠を明記)
   - 【不明】情報不足(何があれば確定できるかを明記)
6. **文書生成**: 下記フォーマットで出力する。

## 出力フォーマット

```markdown
# <システム名> リファレンス(復元版)
## 概要と情報源(何から復元したか・網羅性)
## 外部インターフェース仕様
## 主要な処理フロー・状態遷移
## 設定項目一覧
## 異常系の挙動
## 運用上の注意(日誌由来の暗黙知)
## 未解明事項(不明点と、確定に必要な追加情報)
```

## 注意事項

- 推測で仕様を断定しない。確度ラベル(確実/推定/不明)を必ず付ける。
- 復元文書は「間違っているかもしれない仮説集」として扱われる。反証可能なように
  根拠(ファイル名・行番号・ログ行)を必ず残す。
- 大規模コードは全読みせず、インターフェース→設定→ログ出力箇所の順で優先的に読む。
