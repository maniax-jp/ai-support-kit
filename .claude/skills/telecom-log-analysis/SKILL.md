# 通信設備ログ解析

## 目的

ベンダ独自形式を含む通信設備のログから、異常イベントの抽出・時系列相関・原因推定を行い、
運用者がそのまま使える解析レポートを作成する。

## 手順

1. **形式の把握**: ログの先頭数十行からタイムスタンプ形式・severity 表記・識別子
   (装置名、セルID、MO パス等)の規則を推定する。未知の形式でも既知フォーマットを仮定せず、
   実データから規則を導く。
2. **異常イベントの抽出**: ALARM / MAJOR / MINOR / CRITICAL / WARN / FAILURE / 閾値超過などの
   行を抽出し、アラーム ID・対象オブジェクト・発生値で分類する。
3. **時系列整理**: イベントを時系列に並べ、以下に着目する。
   - 異常の初出時刻と、その直前の状態変化(作業・再起動・ロック解除・設定変更)
   - 同一事象の反復・悪化傾向(rate や件数の推移)
   - 周辺オブジェクトへの波及(隣接セル・対向装置・上位装置)
4. **相関と原因推定**: 利用可能なら他の情報源(オペレータ日誌、パラメータ変更履歴、KPI)と
   突き合わせる。推定には必ず根拠(該当ログ行)を添え、確度(高/中/低)を付す。
5. **レポート作成**: 下記フォーマットで出力する。

## 出力フォーマット

```markdown
# ログ解析レポート
## 対象: <ファイル名> (期間: <開始>〜<終了>)
## サマリ(3行以内)
## 検出イベント一覧(時系列表: 時刻 / severity / 対象 / 内容)
## 相関分析(イベント間・外部情報との関連)
## 原因推定(推定内容 / 根拠となるログ行 / 確度)
## 推奨アクション(調査・復旧・エスカレーション)
```

## 注意事項

- ログにない事実を創作しない。不明な点は「不明」と明記し、追加で必要な情報源を挙げる。
- severity の語彙はベンダごとに異なる。ハードコードされた語彙に頼らず実データから判断する。
- 大容量ログは全読みせず、grep 等で異常行を先に絞り込んでから前後の文脈を読む。
