# 障害報告書作成

## 目的

散在する一次資料(設備ログ、KPI、オペレータ日誌、作業票)から、事実と推定を峻別した
障害報告書を作成する。再発防止に使える具体性を持たせる。

## 手順

1. **資料の突き合わせ**: 提供された全資料からイベントを抽出し、単一のタイムラインに統合する。
   資料間で矛盾があれば隠さず「矛盾あり」として明記する。
2. **影響の定量化**: 影響期間・影響対象(セル/装置/エリア)・ユーザ影響(規模・体感)を
   資料から定量的に示す。定量化できない場合は根拠付きで見積もる。
3. **原因分析**: 直接原因だけでなく、なぜ防げなかったか(検知の遅れ、レビューの欠落、
   手順の不備)まで掘り下げる。人を責める書き方ではなく仕組みの欠陥として記述する。
4. **事実と推定の峻別**: 各記述に根拠資料を紐付ける。推定は【推定】と明記する。
5. **再発防止策**: 「注意を徹底する」のような精神論は書かない。仕組みで防ぐ具体策
   (チェックの自動化、手順の変更、閾値の見直し)を、実施主体と併せて提案する。

## 出力フォーマット

```markdown
# 障害報告書: <件名>
## 概要(発生日時 / 復旧日時 / 影響 / 原因 を各1行)
## タイムライン(表: 日時 / 事象 / 情報源)
## 影響(期間・対象・規模)
## 原因分析(直接原因 / 検知・対応の課題 / 背景要因)
## 対処内容(実施済み / 未実施)
## 再発防止策(表: 施策 / 分類[検知・予防・手順] / 実施主体)
## 未確定事項
```

## 注意事項

- 資料にない事実を創作しない。時刻は必ず資料の記載に基づく。
- 復旧していない事象の報告書では「暫定報告」であることを冒頭に明記する。
- 提供資料の範囲を「情報源」として必ず記載する(後から資料が増えたときの改版のため)。
