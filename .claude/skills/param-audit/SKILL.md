# パラメータ差分監査

## 目的

パラメータダンプ(CSV/JSON/独自形式)の新旧 2 版を比較し、すべての差分を漏れなく列挙して
リスクを評価する。本番同等のテスト環境が用意できない前提のため、投入前の机上監査が
品質確保の最後の砦となる。

## 手順

1. **形式の正規化**: 両ファイルの形式を確認し、キー(オブジェクト識別子)と属性の組に正規化する。
   比較はスクリプト(python3 等)で機械的に行い、目視比較はしない。
2. **差分の全列挙**: 変更・追加・削除されたオブジェクト/属性をすべて列挙する。
   「差分なし」の確認も明示的に行う。
3. **リスク評価**: 各差分について以下の観点で高/中/低を付す。
   - 影響範囲: 単一オブジェクトか、複数・広域か
   - 性質: 電波品質に直結する項目(送信電力、チルト、HO 判定系、再選択系など)か
   - 変化量: 一般的な調整幅を超える大きな変化か
   - 整合性: 関連パラメータ間・隣接オブジェクト間の不整合を生まないか
4. **意図との照合**: 作業票・日誌など変更意図が分かる資料があれば、意図にない差分
   (無断変更・混入)を最優先で指摘する。
5. **変更票の作成**: 下記フォーマットで出力する。

## 出力フォーマット

```markdown
# パラメータ変更監査票
## 比較対象: <旧版> → <新版>
## サマリ(差分件数・最重要リスクを3行以内)
## 差分一覧(表: オブジェクト / 属性 / 旧値 / 新値 / リスク / 根拠)
## 意図外変更の疑い(意図資料がある場合)
## 整合性チェック結果
## 投入可否の見解と条件(承認・追加確認・切り戻し準備)
```

## 注意事項

- 差分の見落としは事故に直結する。必ず機械的比較を行い、件数を明記する。
- リスク評価の根拠を必ず書く。「なんとなく危険」は不可。
- このスキルは監査のみを行う。投入の実行は絶対に行わない(判断と実行の分離)。
