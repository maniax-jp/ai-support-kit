# 作業手順書生成

## 目的

本番同等のテスト環境が存在しない前提で、失敗を持ち込まないための作業手順書を生成する。
すべての手順は「事前確認 → 投入 → 事後確認 → (異常時)切り戻し」の 4 部構成とし、
切り戻し不能な作業を計画段階で検出する。

## 手順

1. **変更内容の確認**: 対象オブジェクト、変更項目、新旧値、変更理由を整理する。
   不明な点は手順書に反映せず、発注者への確認事項として列挙する。
2. **影響分析**: サービス影響(瞬断・再起動の要否)、影響ユーザ規模、影響時間帯を見積もる。
   影響が読めない場合は「読めない」と明記し、最悪ケースを想定した時間帯設定を提案する。
3. **手順の構成**: 以下の構成で作成する。
   - 事前確認: 現状値の取得・保存(切り戻し用)、対象の正常性確認、前提条件
   - 投入: 1 ステップ 1 操作。各ステップに期待結果と NG 時の分岐を付す
   - 事後確認: 設定反映確認、正常性確認(アラーム・主要 KPI)、確認継続期間
   - 切り戻し: 発動判断基準(誰が・何を見て・いつまでに)と、投入と同粒度の手順
4. **安全設計の確認**:
   - 一括投入は段階投入(1 台 → 少数 → 全体)に分割できないか検討し、提案する
   - 切り戻し不能なステップがあれば最上部に警告として明記する
   - 変更前値の取得がない手順は不完全とみなし、必ず追加する
5. **出力**: 下記フォーマットで出力する。

## 出力フォーマット

```markdown
# 作業手順書: <作業名>
## 作業概要(目的 / 対象 / 作業時間帯 / サービス影響)
## 前提条件・事前準備
## 切り戻し可否と発動基準(★最重要)
## 手順
### 1. 事前確認(現状保存を含む)
### 2. 投入(ステップ番号 / 操作 / 期待結果 / NG時の対応)
### 3. 事後確認
### 4. 切り戻し手順
## 確認事項(発注者に確認が必要な不明点)
```

## 注意事項

- 実行コマンドが不明なシステムについてコマンドを創作しない。プレースホルダ
  《要確認: ○○システムの△△コマンド》と明記する。
- 手順書の生成のみを行い、手順の実行は絶対に行わない。
- 「事後確認なし」「切り戻しなし」の手順書は出力しない。
